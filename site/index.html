    1. Кто такой фулстек-аналитик?

    Фулстек-аналитик — это профессионал, который сочетает в себе навыки системного анализа, бизнес-анализа и анализа данных. Он способен работать с полным спектром задач, начиная от выявления потребностей бизнеса, создания документации и проектирования систем, и заканчивая сбором, анализом и визуализацией данных. Это универсальная роль, которая требует знаний в разных областях и инструментов для решения сложных проблем.

    Обязанности фулстек-аналитика

    Фулстек-аналитик выполняет разнообразные задачи, включая:

    • Анализ бизнес-требований: Взаимодействие с заказчиками для выявления проблем и потребностей бизнеса.
    • Проектирование решений: Разработка концепций и архитектуры системы для достижения бизнес-целей.
    • Анализ данных: Сбор и анализ данных для принятия обоснованных решений.
    • Визуализация данных: Создание отчетов и дашбордов для улучшения понимания бизнес-метрик.
    • Работа с базами данных: Написание запросов на SQL для работы с данными.
    • Взаимодействие с командой разработки: Работа с программистами для реализации технической стороны проектов.
    • Отчётность и поддержка: Обеспечение бизнеса необходимыми отчетами и аналитикой.

    Роли и обязанности

    Фулстек-аналитик выполняет задачи, которые объединяют обязанности бизнес-аналитика и системного аналитика. Он также часто работает с данными и применяет инструменты анализа для принятия решений. В отличие от традиционного аналитика, фулстек-аналитик охватывает весь процесс, от сбора данных до их анализа и создания рекомендаций.

    Для кого эта профессия?

    Фулстек-аналитик подходит для тех, кто любит решать сложные задачи, работать с данными и понимать, как технологии помогают бизнесу достигать целей. Это для людей, которые: - Имеют аналитическое мышление. - Обладают интересом к бизнесу и технологиям. - Хотят работать на стыке разных областей (бизнес, технологии, данные). - Желают расширить свои знания и навыки в разных сферах и стать универсальными специалистами.

    Инструменты, которые использует фулстек-аналитик

    Для эффективной работы фулстек-аналитику необходимо владеть рядом инструментов:

    • SQL — для работы с базами данных.
    • Power BI, Tableau — для создания дашбордов и отчетности.
    • Python с библиотеками, такими как Pandas, NumPy — для анализа данных.
    • Excel — для быстрой обработки и анализа данных.
    • Jira и Confluence — для управления проектами и ведения документации.
    • Git и GitHub — для контроля версий и управления кодом.
    • R — для статистического анализа и машинного обучения.
    • BPMN и UML — для моделирования и оптимизации бизнес-процессов.
    • Postman - для тестирования API и работы с веб-сервисами.

    [!NOTE] На самом деле их гораздо больше, но перечислены основные.

    В чем польза фулстек-аналитика для бизнеса?

    Фулстек-аналитик помогает бизнесу:

    • Ускорить процессы: Быстро анализировать и внедрять решения на основе данных.
    • Повысить эффективность: Создавать точные и обоснованные рекомендации по улучшению процессов.
    • Снижение рисков: Прогнозировать результаты на основе анализа данных, снижая вероятность ошибок.
    • Улучшение взаимодействия между командами: Фулстек-аналитик понимает технические и бизнес-требования и может служить связующим звеном между различными департаментами.

    Путь становления фулстек-аналитиком

    Чтобы стать фулстек-аналитиком, вам нужно изучить несколько ключевых областей:

    1. Основы системного анализа — как проектировать и разрабатывать решения для бизнеса.
    2. Бизнес-анализ — как понимать и решать проблемы бизнеса.
    3. Анализ данных — как собирать, обрабатывать и визуализировать данные.
    4. Навыки программирования — изучение языков программирования (Python) и баз данных (SQL).
    5. Инструменты для работы с данными — освоение BI-платформ и инструментов для визуализации и анализа данных.

    Чем фулстек-аналитик отличается от других ролей?

    • Фулстек-аналитик vs Системный аналитик: Системный аналитик обычно сосредоточен на разработке и технической части системы, в то время как фулстек-аналитик охватывает полный цикл, включая анализ данных и создание бизнес-решений.
    • Фулстек-аналитик vs Бизнес-аналитик: Бизнес-аналитик больше фокусируется на выявлении бизнес-требований и их документировании, а фулстек-аналитик помогает не только выявить проблемы, но и разработать решения, включая данные и технологии.
    • Фулстек-аналитик vs Аналитик данных: Аналитик данных работает только с данными, в то время как фулстек-аналитик охватывает более широкий спектр задач, включая бизнес-аналитические аспекты и разработку решений.

    Примеры успешных кейсов

    Фулстек-аналитик может помочь компаниям:

    • Оптимизировать процессы продаж с помощью анализа данных о клиентах.
    • Создать дашборды для отслеживания ключевых показателей бизнеса.
    • Разработать систему для автоматического прогнозирования спроса.
    • Анализировать поведение пользователей на сайте для улучшения пользовательского опыта.

    Почему сейчас так востребован фулстек-аналитик?

    Несмотря на то, что вакансии часто называются «системный аналитик» или «бизнес-аналитик», на самом деле многие работодатели ожидают от кандидатов знаний и навыков в разных областях. Такой универсальный подход становится всё более важным для компаний, стремящихся к оптимизации процессов и более точным данным для принятия решений.

    Фулстек-аналитики способны не только определить и понять потребности бизнеса, но и предложить технические решения, управлять проектами и работать с данными. Это позволяет бизнесу быстро адаптироваться к изменениям на рынке и принимать обоснованные решения на основе данных.

    [!Note] По факту вы не устроитесь на работу, например, только Системным аналитиком. А если и повезет выполянть только определенные для конкретной профессии задачи, то платить вам будут очень мало.

    2. Сбор и анализ требований

    Сбор и анализ требований — это один из самых важных этапов в процессе разработки проекта. Без четко определённых требований можно потратить много времени и ресурсов на создание функций, которые не будут соответствовать ожиданиям заказчика или не будут ему полезны. Этот этап включает в себя несколько ключевых процессов:

    1. Сбор информации

    Сбор информации — это начальный этап, на котором происходит выявление потребностей и проблем заказчика. Важно понять, что именно клиент ожидает от будущего продукта. Основные методы:

    • Интервью: Ключевые пользователи, заказчики или другие заинтересованные стороны должны быть опрошены для получения первичной информации. Интервью могут быть как структурированными (с заранее подготовленными вопросами), так и неструктурированными, где обсуждается широкий круг тем.
    • Опросы и анкеты: Это более формализованный способ получения данных от большой аудитории. Анкеты помогают выявить общее мнение пользователей о том или ином аспекте продукта.
    • Наблюдение: Иногда полезно понаблюдать за текущими процессами работы системы, чтобы понять, как текущая система работает и какие есть проблемы.
    • Анализ существующих данных: Исследование уже имеющихся данных, отчетности и документации, которые могут помочь в формулировке требований.

    Пример: Заказчику нужно мобильное приложение для заказа еды. На этапе сбора информации мы можем провести интервью с пользователями, чтобы понять, какие функции должны быть в приложении (возможность выбора доставки, фильтры по категориям еды).

    2. Анализ собранной информации

    После того как данные собраны, наступает этап анализа и систематизации.

    • Систематизация информации: Важно разделить информацию на несколько категорий, например, функциональные и нефункциональные требования.
      • Функциональные требования: Описание того, что система должна делать (пользователь должен иметь возможность добавлять товар в корзину).
      • Нефункциональные требования: Определяют, как система должна работать (время отклика на запросы не более 3 секунд).

    Инструменты анализа

    • SWOT-анализ: Оценка сильных и слабых сторон, возможностей и угроз в предложенных требованиях.

      • S (Strengths): Сильные стороны (Удобство заказа: Возможность быстрого выбора и оформления заказа).
      • W (Weaknesses): Слабые стороны (Ограниченная география доставки: только несколько городов).
      • O (Opportunities): Возможности (Развитие и расширение списка ресторанов, что улучшит предложения для пользователе).
      • T (Threats): Угрозы (Возможные проблемы с качеством обслуживания или доставкой, что может снизить рейтинг приложения).

    • INVEST: Качества, которые должны быть у хороших требований:

      • Independent (Независимые): Требования не зависят друг от друга.
      • Negotiable (Обсуждаемые): Требования можно обсуждать и изменять.
      • Valuable (Ценные): Требования должны приносить пользу.
      • Estimable (Оценимые): Требования можно оценить по сложности.
      • Small (Малые): Требования должны быть разбиты на небольшие части.
      • Testable (Проверяемые): Требования должны быть проверяемыми через тесты.
    User Story: Как покупатель, я хочу добавлять товары в корзину, чтобы совершать покупки онлайн. Анализ по INVEST:
  1. I (Independent) — Независимые: Требование не зависит от других функций, например, от оформления заказа или системы оплаты. Это значит, что разработчики могут реализовать добавление в корзину отдельно.
  2. N (Negotiable) — Обсуждаемые: Можно обсуждать, какие функции нужны: например, возможность изменять количество товаров в корзине или сохранять корзину после выхода из аккаунта. Требование не должно быть жестко зафиксированным, если можно предложить лучший вариант.
  3. V (Valuable) — Ценные: Без корзины пользователи не смогут оформить заказ, поэтому это критически важная функция. Это требование дает непосредственную пользу пользователю.
  4. E (Estimable) — Оцениваемые: Можно дать примерную оценку трудозатрат на реализацию этой функции (например, 5 дней работы). Если оценить невозможно, значит, требование слишком размыто, и его нужно уточнить.
  5. S (Small) — Малые: Требование описывает конкретную маленькую задачу (добавление товара в корзину), а не весь процесс покупки. Если бы мы сформулировали сразу «создание корзины + оформление заказа + интеграция с платежными системами», это было бы слишком большим требованием.
  6. T (Testable) — Проверяемые: Можно протестировать: добавляется ли товар в корзину, сохраняется ли после обновления страницы, можно ли изменить количество товаров. Если требование нельзя проверить тестами, значит, его сформулировали недостаточно четко.


    • SMART: Условия, которые должны удовлетворять цели:
      • Specific (Конкретные)
      • Measurable (Измеримые)
      • Achievable (Достижимые)
      • Relevant (Релевантные)
      • Time-bound (Ограниченные по времени)
    Анализ по SMART: Улучшение мобильного приложения.
  7. Specific (Конкретность): Улучшить скорость загрузки мобильного приложения.
  8. Measurable (Измеримость): Уменьшить время загрузки с 5 секунд до 2 секунд.
  9. Achievable (Достижимость): Оптимизировать код, сжать изображения, использовать CDN.
  10. Relevant (Релевантность): Быстрое приложение повысит удержание пользователей.
  11. Time-bound (Ограниченность во времени): Реализовать в течение 2 месяцев.


    12. 3. Приоритизация требований

    Когда информация собрана и проанализирована, необходимо понять, какие требования важнее для заказчика и какие из них стоит реализовывать в первую очередь.

    Пример: В приложении доставки еды функция поиска ресторанов будет приоритетнее, чем возможность оставлять отзывы.

    Методы определения приоритетов:

    • Метод MoSCoW: Этот метод позволяет расставить приоритеты:
      • Must have: Обязательные требования (Авторизация пользователя в системе).
      • Should have: Важные, но не обязательные требования(История заказов и повторное оформление).
      • Could have: Желательные требования, которые не критичны для функционирования(Поддержка темной темы интерфейса).
      • Won't have: Требования, которые не будут реализованы в текущей версии(Поддержка нескольких языков).


    • Модель Kano: Оценка потребностей заказчика по степени удовлетворенности от выполнения требования. Разделяется на:
      • Основные: То, что обязательно должно быть (оформление заказа).
      • Ожидаемые: То, что приятно иметь, но не критично (рекомендации блюд).
      • Привлекательные: Необходимые для создания «вау» эффекта, но не обязательно (интерактивные элементы в приложении).

    Пример: В приложении для заказа еды, приоритет может быть отдан функции поиска еды (Must have), а возможность оставлять отзывы о ресторане может быть менее важным, но все равно нужным для пользователей (Should have).

    4. Документирование требований

    После того как требования собраны, проанализированы и приоритизированы, важно все это задокументировать.

    • BRD (Business Requirement Document): Это основной документ, который используется для фиксации всех требований. В нем описываются бизнес-цели, требования пользователей, технические ограничения и ключевые параметры проекта.

    Введение

    • Название проекта: FoodExpress
    • Цель: Разработать мобильное приложение для заказа и доставки еды, которое будет интуитивно понятным и удобным для пользователей.
    • Заказчик: Компания FoodExpress
    • Ответственные за проект: Команда разработчиков, менеджер проекта, дизайнеры, маркетологи.

    Общее описание

    • Приложение должно предоставлять пользователям возможность выбирать блюда, оформлять заказы, отслеживать их статус и выбирать способы оплаты.
    • Основные функции включают регистрацию, создание профиля, добавление товаров в корзину, оформление заказа, оплату, и отслеживание доставки.
    • Приложение будет доступно для платформ iOS и Android.

    Цели и задачи

    • Цель: Повысить уровень комфорта и удобства для пользователей при заказе еды на дом.
    • Задачи:
      • Разработать удобный интерфейс для быстрого оформления заказа.
      • Предоставить возможность выбора ресторана, типа кухни и фильтрации меню.
      • Интегрировать систему онлайн-оплаты.
      • Добавить систему отслеживания заказа в реальном времени.

    Требования

    • Функциональные требования:
      • Возможность регистрации пользователей с помощью email или социальных сетей (Google, Facebook).
      • Поддержка фильтрации и сортировки блюд по категории, цене и времени доставки.
      • Поддержка различных способов оплаты (онлайн и наличные при получении).
      • Поддержка уведомлений о статусе заказа в реальном времени.
    • Нефункциональные требования:
      • Приложение должно работать быстро, с откликом не более 1 секунды.
      • Приложение должно быть доступно 99% времени в месяц.
      • Данные пользователей должны быть защищены с помощью шифрования.

    Ожидаемые результаты

    • Увеличение числа заказов на 30% в течение первого месяца после запуска приложения.
    • Привлечение не менее 5000 активных пользователей в первый месяц работы приложения.
    • Высокая оценка удовлетворенности пользователей, выше 4.5/5.

    Ключевые заинтересованные стороны

    • Заказчик: Компания FoodExpress, ответственная за заказ и финансирование проекта.
    • Разработчики: Команда разработчиков, создающая приложение.
    • Маркетологи: Люди, ответственные за продвижение приложения и привлечение пользователей.
    • Пользователи: Конечные пользователи, которые будут использовать приложение для заказа еды.

    Риски

    • Технические риски: Возможные сбои в платёжных системах и интеграциях с ресторанами.
    • Риски безопасности: Утечка данных пользователей или нарушение конфиденциальности платежной информации.
    • Риски на рынке: Высокая конкуренция в области доставки еды, что может снизить успех проекта.

    Ограничения

    • Приложение не будет поддерживать доставку из супермаркетов в первой версии.
    • Будет ограниченная география, только один город на старте.
    • Первая версия приложения не будет поддерживать мультиязычность.

    Предположения

    • Пользователи будут иметь доступ к интернету для оформления заказов через приложение.
    • Рестораны будут сотрудничать и предоставлять актуальное меню для отображения в приложении.

    Примечания

    • Приложение будет интегрироваться с популярными платёжными системами, такими как PayPal и Stripe.
    • Функция отслеживания заказов будет работать через GPS-систему.
    • FRD (Functional Requirements Document): Детализированное описание функционала системы, включая, как именно система должна выполнять те или иные функции.

    Введение

    • Название проекта: FoodExpress
    • Цель: Создать мобильное приложение для заказа и доставки еды.
    • Краткое описание: Приложение позволит пользователям просматривать меню ресторанов, оформлять заказы, выбирать способы оплаты и отслеживать статус заказа в реальном времени.

    Общее описание

    • Приложение будет доступно на iOS и Android.
    • Пользователи смогут создавать аккаунты, просматривать меню ресторанов, заказывать еду, оплачивать и отслеживать свои заказы.
    • Приложение будет включать возможность оставлять отзывы о ресторанах и блюдах.

    Функциональные требования

    • Регистрация и авторизация:
      • Возможность регистрации с помощью email или через социальные сети (Google, Facebook).
      • Возможность входа в систему с использованием email/пароля или через соцсети.
      • Восстановление пароля через email.
    • Просмотр и фильтрация меню:
      • Отображение списка доступных ресторанов с их меню.
      • Возможность фильтрации по типу кухни (например, пицца, суши, бургер).
      • Возможность сортировать блюда по цене, рейтингу и времени доставки.
    • Корзина и оформление заказа:
      • Возможность добавлять блюда в корзину.
      • Возможность изменения количества или удаления блюд из корзины.
      • Возможность выбора адреса доставки.
      • Возможность выбора способа оплаты (наличные или онлайн).
    • Оплата заказа:
      • Интеграция с платёжными системами для оплаты через карту, Apple Pay, Google Pay.
      • Возможность выбора способа оплаты (картой онлайн или наличными при получении).
    • Отслеживание статуса заказа:
      • Пользователь может видеть статус заказа в реальном времени (например, "готовится", "в пути", "доставлен").
      • Возможность получения уведомлений о статусе заказа через Push-уведомления.
    • Отзывы и рейтинги:
      • Возможность пользователю оставлять отзыв о блюде или ресторане после получения заказа.
      • Возможность оценки блюд по шкале от 1 до 5.
    • История заказов:
      • Возможность просматривать историю всех заказов пользователя.
      • Возможность повторного оформления заказа из истории.

    Нефункциональные требования

    • Производительность:
      • Приложение должно загружаться за 2 секунды или меньше.
      • Время отклика приложения на действия пользователя не должно превышать 1 секунды.
    • Надёжность:
      • Приложение должно быть доступно 99% времени в месяц.
      • В случае сбоя приложения должна быть возможность быстро восстановить работу.
    • Безопасность:
      • Данные пользователя должны быть защищены с использованием шифрования.
      • Вся информация о платёжных данных должна быть хранена в зашифрованном виде и не храниться на сервере.

    Ограничения и исключения

    Успешные критерии

    • В первый месяц работы приложения должно быть оформлено не менее 5000 заказов.
    • Уровень удовлетворённости пользователей должен быть выше 4.5/5.

    Фильтрация ресторанов по рейтингу

    • User Story:
      • Как пользователь,
      • Я хочу фильтровать рестораны по рейтингу,
      • Чтобы видеть заведения с лучшим обслуживанием и качеством еды.
    • Use Case:
      • Акторы: Пользователь
      • Предусловия: Пользователь открыл приложение и находится в каталоге ресторанов.
      • Основной поток:
        • 1. Пользователь открывает фильтр.
        • 2. Выбирает сортировку по рейтингу.
        • 3. Подтверждает выбор.
        • 4. Приложение обновляет список ресторанов, сортируя их по убыванию рейтинга.
      • Альтернативные потоки:
        • 2а. Пользователь выбирает сортировку по возрастанию рейтинга.
        • 2б. Пользователь отменяет выбор — список остаётся без изменений.
      • Результат: Пользователь видит рестораны, отсортированные по рейтингу.
      • Критерии приемки:
        • Система должна отсортировать рестораны по рейтингу при применении фильтра.
        • Если сортировка не производится корректно, должно отображаться сообщение об ошибке.
        • Рестораны должны быть отсортированы от лучшего к худшему или наоборот в зависимости от выбора пользователя.

    Добавление блюда в корзину

    • User Story:
      • Как пользователь,
      • Я хочу добавить блюдо в корзину одним нажатием,
      • Чтобы быстро формировать заказ без лишних действий.
    • Use Case:
      • Акторы: Пользователь
      • Предусловия: Пользователь просматривает меню ресторана.
      • Основной поток:
        • 1. Пользователь нажимает кнопку «Добавить в корзину».
        • 2. Приложение увеличивает счётчик товаров в корзине.
        • 3. Приложение отображает всплывающее уведомление об успешном добавлении.
      • Альтернативные потоки:
        • 2а. Пользователь удаляет блюдо из корзины.
        • 3а. Приложение показывает сообщение об ошибке (если произошёл сбой).
      • Результат: Блюдо добавлено в корзину, пользователь может продолжить заказ.
      • Критерии приемки:
        • После нажатия кнопки «Добавить в корзину» блюдо должно появиться в корзине.
        • При сбое в процессе добавления должно быть отображено сообщение об ошибке.
        • Количество товаров в корзине должно увеличиться.

    Отслеживание заказа

    • User Story:
      • Как пользователь,
      • Я хочу отслеживать статус заказа в реальном времени,
      • Чтобы знать, когда мне его доставят.
    • Use Case:
      • Акторы: Пользователь
      • Предусловия: Пользователь оформил заказ.
      • Основной поток:
        • 1. Пользователь открывает раздел «Мои заказы».
        • 2. Приложение загружает информацию о статусе заказа.
        • 3. Пользователь видит текущее местоположение курьера и ожидаемое время доставки.
      • Альтернативные потоки:
        • 2а. Если статус заказа не обновляется, приложение показывает сообщение о возможной задержке.
      • Результат: Пользователь получает актуальную информацию о заказе.
      • Критерии приемки:
        • Информация о заказе должна обновляться в реальном времени.
        • При отсутствии обновлений статус должен быть отмечен как «ожидает обновления» или аналогично.

    Оплата онлайн

    • User Story:
      • Как пользователь,
      • Я хочу оплачивать заказ картой или через электронные кошельки,
      • Чтобы не тратить время на расчёт наличными.
    • Use Case:
      • Акторы: Пользователь
      • Предусловия: В корзине есть товары, пользователь готов оформить заказ.
      • Основной поток:
        • 1. Пользователь нажимает кнопку «Оформить заказ».
        • 2. Приложение предлагает выбрать способ оплаты.
        • 3. Пользователь выбирает онлайн-оплату и вводит данные карты.
        • 4. Приложение обрабатывает платёж и подтверждает оплату.
      • Альтернативные потоки:
        • 3а. Пользователь выбирает оплату наличными.
        • 4а. Оплата не проходит, приложение предлагает повторить попытку или выбрать другой способ.
      • Результат: Заказ успешно оплачен, пользователь получает подтверждение.
      • Критерии приемки:
        • Платёж должен быть обработан корректно и отображён статус успешной оплаты.
        • В случае неудачной попытки оплаты пользователь должен получить сообщение об ошибке и опцию повторить оплату.

    Сохранение избранных ресторанов

    • User Story:
      • Как пользователь,
      • Я хочу сохранять любимые рестораны в избранное,
      • Чтобы быстро находить их и заказывать снова.
    • Use Case:
      • Акторы: Пользователь
      • Предусловия: Пользователь просматривает список ресторанов.
      • Основной поток:
        • 1. Пользователь нажимает на значок «Добавить в избранное».
        • 2. Приложение сохраняет ресторан в список избранного.
      • Альтернативные потоки:
        • 2а. Пользователь удаляет ресторан из избранного.
      • Результат: Ресторан добавлен в избранное, доступен для быстрого выбора.
      • Критерии приемки:
        • Ресторан должен быть добавлен в список избранных после нажатия на кнопку.
        • После добавления ресторан должен быть доступен в разделе «Избранное».

    5. Минимально жизнеспособный продукт

    Minimum Viable Product (MVP): Концепция, которая подразумевает создание минимально жизнеспособной версии продукта, которая содержит только самые необходимые функции для того, чтобы продукт был полезен для пользователей и мог быть выпущен в тестирование или на рынок. MVP основывается на приоритетных требованиях, которые дают максимально возможную ценность для пользователей на первом этапе.
